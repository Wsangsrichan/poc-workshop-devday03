# ใบให้คะแนน (แลปสแกนความปลอดภัยโค้ด)

ให้คะแนนแต่ละทีมโดยดู “หลักฐาน” (รายงาน + คำอธิบาย) ไม่ใช่แค่รันคำสั่งผ่าน ๆ

หลักฐานอาจเป็นไฟล์รายงาน JSON, ภาพหน้าจอ, หรือการคัดลอก finding ที่มี path + เลขบรรทัดชัดเจน  
คำแนะนำสำหรับการส่งงาน: ส่งรายงานล่าสุด 3 ไฟล์ + โน้ตสั้น ๆ ว่า 1 false positive คืออะไร และแก้อะไรไปแล้ว 3 จุด

## Part A: Semgrep (10 คะแนน)
- [ ] (2) มีรายงาน Semgrep JSON อยู่ใน `reports/` (ตัวอย่าง: `reports/<target>.latest.semgrep.json`)
- [ ] (3) แยกผลตรวจพบตามความรุนแรง และยกตัวอย่างอธิบายอย่างน้อย 1 รายการต่อระดับ
- [ ] (3) ระบุ false positive อย่างน้อย 1 รายการ (หรืออธิบายว่าทำไมไม่มี)
- [ ] (2) เปรียบเทียบผลจาก tool กับการรีวิวด้วยตาเปล่า

## Part B: Syft + Dependency-Track (10 คะแนน)
- [ ] (3) มีไฟล์ SBOM แบบ CycloneDX จาก Syft อยู่ใน `reports/` (ตัวอย่าง: `reports/<target>.latest.sbom.cyclonedx.json`)
- [ ] (3) ระบุ component ที่เสี่ยงที่สุด (CVE/CVSS) จาก Dependency-Track (หรืออธิบายข้อจำกัดถ้าไม่มีเซิร์ฟเวอร์)
- [ ] (2) เสนอเวอร์ชันที่ควรแก้/อัปเกรด หรือแนวทางลดความเสี่ยง
- [ ] (2) อธิบายแนวคิด policy gate (จะบล็อกอะไรใน CI)

## Part C: Gitleaks (10 คะแนน)
- [ ] (3) มีรายงาน Gitleaks JSON อยู่ใน `reports/` (ตัวอย่าง: `reports/<target>.latest.gitleaks.json`)
- [ ] (3) แยกผลตรวจพบที่เป็น “ของจริง” กับ “ผลลวง” (อย่างน้อย 2 รายการ)
- [ ] (2) เสนอขั้นตอนแก้ไข (เช่น ย้ายไป env vars / rotate creds)
- [ ] (2) เพิ่ม/แก้ allowlist ใน `configs/.gitleaks.toml` (เฉพาะกรณีที่มีเหตุผลรองรับ)

## แบบฝึกหัด: แก้ไข + ยืนยันผล (20 คะแนน)
- [ ] (10) แก้ปัญหา 3–5 รายการที่สำคัญ (แสดง diff + เหตุผล)
- [ ] (10) สแกนซ้ำแล้วพิสูจน์ว่า issue ลดลง/ถูกแก้แล้ว (เทียบ before/after)
